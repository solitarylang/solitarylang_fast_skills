# 执行不合理判定规则

## 目的

在运行证据采集完成后，用来判断当前 Spark 执行是否存在不合理现象。

## 什么叫“不合理”

如果证据显示以下任意一种情况，就可以认为执行不合理：

1. job 或 stage 的运行时长相对于当前数据规模明显偏长。
2. job 或 stage 重复做了同一批扫描、join、聚合或写入，而看不出复用收益。
3. shuffle read/write、spill 或 GC 相对于最终有效输出明显偏大。
4. executor 出现失败、抖动或重试，且已经明显影响执行稳定性。
5. task 时间分布出现明显长尾或不均衡。
6. 执行链路里存在 action 或额外触发，导致完整 lineage 被重复执行，但看不出明确业务必要性。
7. 运行过程中生成了大量中间行或中间列，最后又被丢弃。
8. 出现小文件压力、分区过碎，或者 shuffle stage 明显过大。
9. Spark 参数和运行日志明显冲突，例如 executor memory 很大但频繁 spill / GC，executor 数很少但 stage 并行度明显不足，shuffle partitions 明显过粗或过碎。

## 当前判定维度

每个 run 都按下面几个维度检查：

- 运行时长与输入规模是否匹配
- shuffle 体量与有效输出是否匹配
- 重试 / 失败 / executor 抖动是否明显
- task 是否均衡，是否存在长尾
- 是否存在重复扫描或重复 action
- 是否存在无用中间数据
- 文件大小和分区形态是否合理

## 输出格式

每个不合理候选都记录：

- 不合理点是什么
- 哪些证据支持它
- 它是已确认还是仅候选
- 它为什么会影响后续优化

如果当前看不出明显不合理，就直接写成 `task_health_label` 风格或 `normal`，前提是现有证据足够。

## 场景扩展槽位

后续可以直接在这里追加场景规则，不用改流程：

- shuffle 很大但输出很小
- executor 丢失导致 stage 重试
- 热点 key 导致的 skew
- `count()` 类 action 触发的重复 lineage
- 未使用特征分支造成的浪费
- 小文件主导的读写模式
- Spark 参数与日志冲突导致的调参候选
