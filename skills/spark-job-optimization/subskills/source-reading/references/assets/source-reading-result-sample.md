# 源码阅读结果样例框架

这个样例展示 `source-reading` 的最终输出结构，后续所有源码阅读结果都建议按这个要素集和顺序输出。

> 说明：这里的内容是“结果样例框架”，不是运行结论，也不依赖 Spark UI。

## 1. 输入摘要

输入不需要固定模板，只要把下面这些元素说清楚即可：

- 源码说明文档：`sample_source_doc.md`
- 源码目录：`repo/jobs/sample_feature/`
- 主类 / 主函数：`SampleFeatureJob.main`
- 传入参数：`input_dt`, `output_table`, `db_prefix`

## 2. 入口与主链路

### 2.1 入口定位

- 入口文件：`sample_feature_job.py`
- 主类 / 主函数：`SampleFeatureJob.main`
- 调用方式：`main(input_dt, output_table, db_prefix)`

### 2.2 主链路说明

1. 读取输入表 `dwd_event`
2. 裁剪字段并做基础过滤
3. 关联维表 `dim_item`
4. 聚合出用户粒度指标
5. 计算衍生指标并写出结果表

## 3. Stage 级源码对照表

| Stage | 代码范围 | 核心动作 | 关键算子 | 状态 |
|---|---|---|---|---|
| Stage 1 | `10-28` | 输入读取、字段裁剪、基础过滤 | `select -> filter -> withColumn` | `已确认` |
| Stage 2 | `30-52` | 维表关联与聚合 | `join -> groupBy -> agg` | `待确认` |
| Stage 3 | `54-68` | 衍生指标计算与写出 | `withColumn -> fillna -> write` | `已确认` |

## 4. `源码 -> stage -> 指标` 表

| 源码范围 | Stage | 指标 | 说明 |
|---|---|---|---|
| `10-28` | Stage 1 | 输入行数、过滤后行数、字段裁剪效果 | 重点判断是否存在无关列传递 |
| `30-52` | Stage 2 | shuffle 读写、任务数、聚合耗时、长尾 | 重点判断 join / groupBy 代价 |
| `54-68` | Stage 3 | 写出耗时、输出大小、末端整理开销 | 重点判断是否还有冗余列参与写出 |

## 5. Stage 细项卡片

每张 stage 卡片都对应一段连续的算子块，边界必须和上面的 stage 表一致。

### Stage 1

| 项目 | 内容 |
|---|---|
| 代码范围 | `10-28` |
| 算子链 | `scan -> select -> filter -> withColumn` |
| 作用 | 读取输入表并完成基础清洗 |
| 风险点 | 输入字段未裁剪、过滤前置是否充分 |
| 观察项 | 是否存在下游未消费的中间列 |
| 状态 | `已确认` |
| 证据缺口 | 需要后续运行证据确认过滤后数据量变化 |

### Stage 2

| 项目 | 内容 |
|---|---|
| 代码范围 | `30-52` |
| 算子链 | `join -> groupBy -> agg` |
| 作用 | 关联维表并聚合出目标粒度 |
| 风险点 | shuffle 放大、是否存在倾斜、是否需要中间态复用 |
| 观察项 | 关联键是否可以广播、聚合前是否还能继续裁剪 |
| 状态 | `待确认` |
| 证据缺口 | 需要后续运行证据补齐 shuffle read / write 和任务分布 |

### Stage 3

| 项目 | 内容 |
|---|---|
| 代码范围 | `54-68` |
| 算子链 | `withColumn -> fillna -> write` |
| 作用 | 计算衍生指标并落表 |
| 风险点 | 末端整理是否还保留冗余列 |
| 观察项 | 写出前是否还有可裁剪字段 |
| 状态 | `已确认` |
| 证据缺口 | 一般无硬性缺口，除非写出异常 |

## 6. 关键源码信号

| 分类 | 信号 | 处理方式 |
|---|---|---|
| 入口与链路 | 主函数、主数据流、分支链路 | 先画清入口和主链路 |
| 重复计算 | 重复读取、重复 join、重复聚合 | 标成重复执行候选 |
| 列级变换 | `select`、`withColumn`、`drop`、`alias` | 检查冗余中间列是否下游消费 |
| 增行点 | `explode`、`flatMap`、`posexplode` | 单独作为膨胀热点标记 |
| Action 点 | `count()`、`show()`、`collect()` | 判断是否触发全量执行 |
| 大扫描 / 大分区 | 长历史扫描、窗口排序、join、groupBy | 先判断总量还是倾斜 |
| 中间态复用 | scan 后很快收缩到 shuffle 前 | 优先考虑中间表或中间快照 |
| 小文件问题 | 文件数多、单文件小 | 标成小文件风险候选 |
| 冗余字段 | 下游未消费且未写入 | 优先删除计算逻辑 |

## 7. 输出结论

| 结论项 | 内容 |
|---|---|
| 已确认 | Stage 1 和 Stage 3 的代码边界稳定，可直接产出结构化结果 |
| 待确认 | Stage 2 的 shuffle / 倾斜 / 长尾需要运行证据补齐 |
| 需要补齐的证据缺口 | Stage 2 的 shuffle 指标、任务分布、是否存在长尾 |

## 8. 固定产出要求

后续每次输出都应至少包含：

1. 输入摘要
2. 入口与主链路
3. Stage 级源码对照表
4. `源码 -> stage -> 指标` 表
5. Stage 细项卡片
6. 关键源码信号
7. 输出结论
8. 固定的证据缺口列表

如果某个 case 信息更少，可以缩短文字，但不要删掉这些要素。
