# 问题定位与优化建议

## 输入契约

### 必需输入

- 第 1 步输出的结构化结果
- 第 2 步输出的结构化结果

### 可选输入

- 当前代码约束
- 用户允许的改动范围
- 历史 case 经验

### 输入要求

- 输入必须能映射到至少一个 stage 和一个代码范围
- 如果缺少第 1 步或第 2 步的关键证据，必须标记 `待确认`
- 输入需要保留结构化字段，不接受只剩自然语言摘要的材料

## 输出契约

### 必需输出

- 前五个根因排序
- 每个根因对应的 stage 和代码范围
- 每个根因的证据、影响度和待确认项
- 参数调优候选
- 实现方式优化
- 业务逻辑优化
- `优化项 -> 代码片段 / 代码草图 -> 目标 stage -> 运行证据 -> 预期收益` 映射表

### 输出要求

- 第 3 章和第 4 章都应能直接使用
- 优化顺序必须是 `Spark 参数 -> 实现方式 -> 业务逻辑`
- 不能把观察项混进主瓶颈列表
- 默认先落盘到 `<business_repo_root>/tmp/spark-job-optimization/<case_name>/tmp/`，不要写到 skill 仓库或当前工作目录

## 非目标

- 不做代码实际变更
- 不做第 6 步前后对比
- 不替代父 skill 的最终报告编排

## 执行流程

1. 读取来自第 1 步和第 2 步的结构化结果。
2. 把症状对齐到 stage 和代码范围。
3. 对根因按影响度、证据强度和可修复性排序。
4. 把根因映射成可执行的优化项，按 `Spark 参数 -> 实现方式 -> 业务逻辑` 顺序输出。
5. 给出待确认项、风险、预期收益和代码草图。

## 验收标准

- 能独立完成根因排序和优化建议输出
- 能明确区分已确认与待确认
- 输出可直接进入最终报告第 3 章和第 4 章
